## 1. Как устроен Cursor

Cursor — IDE с AI-агентом, который:
- читает файлы в открытой папке (workspace);
- правит код, запускает команды в терминале;
- ищет по кодовой базе;
- **не помнит прошлые чаты автоматически**.

### Три способа «научить» агента вашему проекту

| Способ | Где | Когда подгружается |
|--------|-----|-------------------|
| **User Rules** | Settings → Rules | Каждый чат (глобально) |
| **Project Rules** | `.cursor/rules/*.mdc` | По правилам файла (always / glob / @) |
| **Memory Bank** | `memory-bank/*.md` | Когда вы `@`-упомянете или агент прочитает |
| **Код** | `worker.py`, и т.д. | Когда открыт, `@`-упомянут или найден поиском |

### Главная формула

```
Новый чат = чистый лист
Память = файлы в git + rules
Не память = история чатов (локально на каждом ПК)
```

## 2. Модели: что выбрать

В чате/агенте можно выбрать модель (Settings → Models).

| Модель | Для чего | Скорость / цена |
|--------|----------|-----------------|
| **Composer 2.5 Fast** | Ежедневный кодинг в чате | Быстро, дороже (~$3/M input) |
| **Composer 2.5 Standard** | Фоновые / Cloud Agent задачи | Медленнее, дешевле (~$0.5/M input) |
| **GPT-5.3 Codex** | Сложный рефакторинг, архитектура | Frontier coding |
| **GPT-5.5 Medium** | Планирование, объяснения | Универсальная |
| **Claude Opus / Sonnet** | Если включены в настройках | По вкусу |

### Composer 2.5 Fast — что это

- **Не отдельная модель**, а переключатель скорости у Composer 2.5.
- Fast = низкая задержка, удобно в живом диалоге (**дефолт**).
- Standard = дешевле, для задач «запустил и жду».
- Переключение: model picker → навести на Composer 2.5 → Edit → toggle Fast, или `Ctrl+Alt+/`.
- Окно контекста Composer 2.5: **фиксировано 200k** (Max Mode не расширяет).

### Что брать для AI_MESH

| Задача | Модель |
|--------|--------|
| Правки worker, мелкий код | Composer 2.5 Fast |
| Большой рефакторинг pipeline | Codex или Composer Standard |
| Архитектура (RunPod, API, S3) | Plan mode + GPT-5.5 или Codex |
| Объяснить код брату / себе | Composer Fast или GPT-5.5 |

## 3. Контекст и Context Used

### Что такое «контекст»

Всё, что модель видит **в текущем запросе**:

- история **этого** чата;
- файлы через `@file`, `@folder`;
- результаты поиска по репо;
- вывод терминала, ошибки, скриншоты;
- Project Rules (`alwaysApply: true` и glob-rules);
- User Rules;
- системные инструкции Cursor.

### Context Used

Счётчик в UI чата — сколько токенов из лимита занято.

Ориентиры:
- 1 токен ≈ 3–4 символа текста;
- **200k токенов** ≈ много текста, но **не весь репозиторий**;
- лимит расходуется быстрее, чем кажется.

### Что съедает контекст быстрее всего

1. Длинные чаты (вся история уходит в каждый запрос).
2. `@` на большие папки (например, весь `TRELLIS/`).
3. Огромные rules с `alwaysApply: true`.
4. Длинные логи, JSON-дампы, base64 в чате.

### Когда контекст переполняется

- Cursor сжимает (summarize) старые сообщения;
- или просит новый чат;
- качество падает — «забываются» детали из начала разговора.

### Практика для большого проекта

- Прикрепляй только нужные файлы.
- Дроби задачи: worker / API / frontend отдельно.
- Стабильные знания — в **memory-bank**, не в бесконечный чат.
- Rules — **короткие** (до ~50 строк на alwaysApply).

```
┌─────────────────────────────────────┐
│  Бюджет 200k токенов на один запрос │
├─────────────────────────────────────┤
│  Rules (alwaysApply)                │
│  + История чата                     │
│  + @ файлы                          │
│  + Вывод tools / терминал           │
│  + Системный prompt                 │
└─────────────────────────────────────┘
           ↓
    Context Used растёт
```

## 4. Память: Obsidian vs Cursor

### Встроенного Obsidian vault в Cursor нет

Но есть эквивалент:

| Механизм Cursor | Аналог Obsidian |
|-----------------|-----------------|
| User Rules | Личные заметки |
| `.cursor/rules/` | Wiki проекта |
| `memory-bank/` | Ежедневник + vault |
| `@файл.md` | Ссылка на заметку |
| Memories (через UI) | Закладки |
| История чатов | **Не vault** — локальный архив |

### Рекомендуемая схема (лучше чистого Obsidian)

1. **Код и доки — в git** (единый источник правды).
2. **`.cursor/rules/`** — стабильные правила (стек, RunPod, форматы).
3. **`memory-bank/`** — живой статус и решения.
4. **Obsidian опционально** — если удобно писать, синхронизируй с `docs/` или `memory-bank/`, но агенту указывай пути **внутри workspace**.

### Чего избегать

- Rule на 500 строк с `alwaysApply: true` — съест контекст каждый чат.
- «Он же помнит вчера» — **не помнит**.
- Одно и то же в Obsidian, rules и README — со временем разъедется.

## 5. Структура memory-bank в нашем проекте

```
paradox_worker/
├── .cursor/rules/
│   ├── project-core.mdc       ← alwaysApply: язык, память, git
│   └── runpod-worker.mdc      ← при worker.py, Dockerfile
├── memory-bank/
│   ├── projectbrief.md        ← ЗАЧЕМ проект (редко меняется)
│   ├── techContext.md         ← стек, API, RunPod, секреты
│   ├── systemPatterns.md      ← pipeline, архитектурные решения
│   ├── activeContext.md       ← ЧТО СЕЙЧАС (каждую сессию!)
│   ├── teamWorkflow.md        ← синхронизация вдвоём
│   └── cursor-shpargalka.md   ← этот файл
├── scripts/
│   ├── sync-start.ps1         ← git pull + превью контекста
│   └── sync-end.ps1           ← чеклист перед push
└── AGENTS.md                  ← точка входа для агента
```

### Что в каком файле

| Файл | Содержание | Кто обновляет | Как часто |
|------|------------|---------------|-----------|
| `projectbrief.md` | Цель платформы AI_MESH, scope MVP | Вы | Раз в месяц |
| `techContext.md` | API `{ image_url }` → `model_base64`, RunPod, Docker | Агент / вы | При смене API |
| `systemPatterns.md` | Symlink ckpts, base64 GLB, pipeline | Агент / вы | При решениях |
| `activeContext.md` | Чеклисты, журнал сессий, следующий шаг | **Каждую сессию** | Ежедневно |
| `teamWorkflow.md` | Git sync вдвоём | Редко | При смене процесса |

### Язык

- Memory-bank — **русский** (удобно вам).
- Код и комментарии — **английский**.
- Агенту всё равно на язык файлов — читает одинаково хорошо.

## 6. Рабочий день: практика

### Утро — начало сессии

```powershell
cd D:\AI_HUB\paradox_worker
.\scripts\sync-start.ps1
```

**Новый чат** в Cursor (не продолжай вчерашний!):

```
@memory-bank/activeContext.md

Продолжаем. Кратко: где остановились и что делаем сегодня?
```

### День — работа

Прикрепляй только нужное:

```
@memory-bank/activeContext.md @worker.py

Добавь параметр seed в job input
```

Для архитектуры:

```
@memory-bank/systemPatterns.md @memory-bank/techContext.md

Как лучше подключить сайт к RunPod — runsync или async?
```

### Вечер — конец сессии

В чате:

```
Обнови memory-bank/activeContext.md:
- журнал сессий (кто, что сделано)
- чеклисты
- следующий шаг
Не коммить — я сам push.
```

Терминал:

```powershell
.\scripts\sync-end.ps1
git add memory-bank/ worker.py   # что менялось
git commit -m "Add seed param; update activeContext"
git push
```

### Неделя — типичный цикл (сайт + worker)

| День | Задача | @ в чате | Обновить после |
|------|--------|----------|----------------|
| Пн | Аудит worker | `@worker.py` `@activeContext` | activeContext |
| Вт | API route (будущий сайт) | `@techContext` | techContext + activeContext |
| Ср | Upload UI | `@activeContext` | activeContext |
| Чт | Three.js viewer | `@systemPatterns` | systemPatterns |
| Пт | Деплой | `@techContext` | techContext |

## 7. Кто обновляет память — ты или агент

### Оба варианта OK — лучше гибрид

| | Агент (по запросу) | Руками |
|--|-------------------|--------|
| **Когда** | Конец сессии с кодингом | Мелкая правка, решение без чата |
| **Плюс** | Пишет по факту сделанного | Полный контроль |
| **Минус** | Нужен явный запрос | Нужно помнить формат |

### Промпт для агента

```
Обнови activeContext: мы добавили seed, тест OK. Следующий — API route.
Кто работал: [имя].
```

### Что обновлять когда

| Событие | Файл |
|---------|------|
| Закончили задачу | `activeContext.md` |
| Приняли архитектурное решение | `systemPatterns.md` |
| Поменяли API / стек | `techContext.md` |
| Поменялась цель проекта | `projectbrief.md` |

Агент **предложит** обновить activeContext после задачи (правило в `project-core.mdc`).

## 8. Синхронизация вдвоём на разных ПК

### Один аккаунт Cursor ≠ общие чаты

| Синхронизируется | Нет |
|------------------|-----|
| Подписка PRO | История чатов |
| memory-bank через **git** | Локальные чаты |
| `.cursor/rules/` через git | `.env` (локально у каждого) |
| Код через git | User Rules (ненадёжно) |

### Золотое правило команды

```
Начало:  git pull  →  @activeContext  →  работа
Конец:   обновить activeContext  →  git push
```

### Первый раз (брат)

```powershell
git clone https://github.com/SatanExist/paradox_worker.git
cd paradox_worker
# .env с RUNPOD_API_KEY — локально, не в git
.\scripts\sync-start.ps1
```

### Пример: два дня, два человека

**День 1 — ты:**
1. `sync-start.ps1`
2. Чат: добавить `seed` в worker
3. «Обнови activeContext»
4. `git push`

**День 2 — брат:**
1. `sync-start.ps1` — видит твой журнал
2. **Новый чат**: `@activeContext` — продолжает без твоего диалога
3. Работа → push

**День 3 — ты:**
1. `git pull` — видишь запись брата

### Конфликт в activeContext

```powershell
git pull
# CONFLICT in memory-bank/activeContext.md
```

Открыть файл, **объединить обе записи** в журнале сессий, убрать маркеры `<<<<<<<`, commit, push.

### Общие правила — в git, не только User Rules

Всё важное для команды — в `.cursor/rules/project-core.mdc`, чтобы после `git pull` было одинаково у обоих.

Подробнее: `memory-bank/teamWorkflow.md`

## 9. Как сохранить диалог

### Автоматически

Чат уже в **истории Cursor** (боковая панель). Переименуй вкладку, чтобы найти потом.

### Экспорт

Меню чата `...` → **Export Chat** → Markdown.

### Для проекта (рекомендуется)

Не полагаться на чаты — **memory-bank + git push**.

Опционально для важных решений:
- экспорт чата в `docs/chats/2026-07-08-onboarding.md` (редко).

### Массовый экспорт чатов (опционально)

- Расширение Cursor Chat Exporter
- CLI `cursor-history` / `cursaves`

Для пары разработчиков обычно **не нужно** — хватает memory-bank.

## 10. Режимы Cursor

| Режим | Когда |
|-------|-------|
| **Agent** | Кодит, запускает команды, правит файлы |
| **Ask** | Только вопросы, без изменений |
| **Plan** | Сначала план, потом реализация (архитектура, большие задачи) |

### @-упоминания

| Синтаксис | Что делает |
|-----------|------------|
| `@memory-bank/activeContext.md` | Подгружает файл |
| `@worker.py` | Конкретный файл |
| `@folder` | Папка (осторожно — много токенов) |
| `@Codebase` | Поиск по репо |

## 12. Частые ошибки

| Ошибка | Последствие | Решение |
|--------|-------------|---------|
| Один чат на неделю | Context Used переполнен | Новый чат + `@activeContext` |
| Не делали push | Брат не видит прогресс | `sync-end` → push |
| Ждут общих чатов на одном аккаунте | Разочарование | memory-bank + git |
| `@TRELLIS/` целиком | Сжигает контекст | Только нужные файлы |
| API key в memory-bank | Утечка | Только `.env`, gitignore |
| Дублируют всё в Obsidian и git | Расхождение | Один источник — репо |
| Не pull перед работой | Конфликты, устаревший контекст | `sync-start.ps1` |

## 13. Карта файлов paradox_worker

### Наш MVP сегодня

```
Клиент (test_req.py или будущий сайт)
    │  POST /runsync { input: { image_url } }
    ▼
RunPod Serverless → worker.py
    │  TRELLIS-image-large
    │  render_glb → base64
    ▼
{ status, message, model_base64 }
```

### Ключевые файлы кода

| Файл | Назначение |
|------|------------|
| `worker.py` | RunPod handler |
| `Dockerfile` | CUDA 11.8, TRELLIS, FlexiCubes |
| `test_req.py` | Локальный тест API |
| `TRELLIS/` | Vendored исходники (не трогать без нужды) |
| `.env` | `RUNPOD_API_KEY` — **не в git** |

### RunPod — не ломать

- `HF_HOME=/runpod-volume/huggingface_cache`
- Веса: `/runpod-volume/trellis-weights`
- Symlink: `/app/ckpts` → `{model_path}/ckpts`
- Lazy `load_model()` — один раз на процесс

### Быстрый тест

```powershell
cd D:\AI_HUB\paradox_worker
python test_req.py
```

Ожидание: `"status": "success"` + длинный `"model_base64"`.

| Проблема | Действие |
